# 官方 3.5.6 深度复刻对比报告

更新时间：2026-05-29

对比对象：

- 官方反编译目录：`E:\test\tlcq\_____3.5.6`
- 当前 Flutter 项目：`E:\aaatest\tailg-ble-app`

## 总体结论

当前项目已经不是单纯页面仿制，核心控车链路已覆盖到 BLE/QGJ 协议、官方云登录、官方车辆列表和基础云端控车。但官方 3.5.6 是完整的“账号云端 + 车辆绑定 + 本地/远程控车 + 地图轨迹 + 服务支付 + 多车系硬件生态”应用，当前项目仍是一个聚焦控车和安全复刻边界的子集。

按官方完整 App 口径估算，当前复刻度约 **25%-35%**。按“本地 BLE 控车工具 + 官方云控车入口”口径估算，当前复刻度约 **65%-75%**。

## 已复刻得比较好的部分

- QGJ 本地 BLE 初始化和保活：官方连接后会请求大 MTU、订阅 `feb2`、订阅 `fcc0` 相关特征、存在 `fe01` 时订阅 `fe03`，登录后轮询 `feb3`；当前项目已按这个方向实现。
- 基础控车闭环：本地 BLE 和官方云端都已支持设防、解防/解锁、上电、断电、寻车、开坐垫。
- 骑行模式：当前按官方 `fcc1State2` 低 3 位读写，并保留其他位，避免误改 ECU 状态。
- QGJ 设置命令：光感、声音、震动灵敏度已对齐官方命令；高级命令保持只读或禁写，安全边界正确。
- 官方云第一批：验证码、登录、车辆列表、车辆状态、云端控车、自检、官方车辆与本地 BLE 关联已经形成基本闭环。
- 诊断和日志：BLE/操作日志、诊断报告、官方云请求摘要、GATT 清理等对真实调试有价值。

## 差距最大的部分

1. **官方绑定体系**

   当前只有本地 BLE 车库和官方车辆关联。本质上还没有复刻官方绑定链路，包括扫码、IMEI、门店、重复绑定、解绑限制、控制器激活限制等。

2. **云端数据生态**

   当前官方云只接了账号、车辆状态、基础控车、自检。官方剩余接口覆盖消息、轨迹、围栏、BMS、NFC、分享、服务、门店、SIM、支付等，尚未形成官方 App 级数据体验。

3. **NFC 真钥匙**

   官方存在 `getNFCByQgj` / `setNFCByQgj`，并区分出厂钥匙和用户钥匙，最多 8 个用户钥匙。当前只是本地占位，不会读取或写入车辆钥匙。

4. **BMS/TLV 实读**

   当前 BMS 页面字段结构对了，但数据来源还不完整。官方有 `app/mine/bmsBatteryInfo`、`app/mine/bmsTlvBatteryInfo`、`app/car/bmsTlvExt/report` 和 `app/centralControl/bmsTlvBatterySetUp` 等接口/流程。

5. **地图轨迹围栏**

   当前只有本地坐标和外部地图跳转。官方有完整地图、历史轨迹、轨迹详情、轨迹回放、电子围栏配置和云端定位服务。

6. **多车系硬件**

   当前重点是 QGJ 和标准 BLE。官方还覆盖 BB/贝欧、JLFBXW/C39、TPMS、GPS、报警器、控制器等多个硬件体系。

7. **服务/支付/续费生态**

   官方有充电站、订单、支付、流量包、续费、售后、门店、保险等。当前未接入，且不建议作为短期主线。

## 不建议直接复刻的边界

以下能力即使官方存在，也不建议直接开放，除非已经完成反编译证据、真机验证、失败回滚和用户确认：

- NFC 真加钥匙、强制删除钥匙、恢复钥匙。
- 远程/本地 OTA 写入，包括 QGJ、报警器、控制器、GPS、C39、V3。
- 密码解锁读取和写入，尤其可能返回或修改敏感密码内容的流程。
- 自动锁车、电子龙头锁、边撑、坐垫、侧翻检测等高级 ECU 写入。
- 支付、续费、充电桩交易、保险、商城、金融服务。
- 绕过官方账号、官方 token 或官方车辆绑定关系的能力。

## 下一批建议

### 可以优先做

- 官方 BMS/电池详情只读：接入官方 `bmsBatteryInfo` / `bmsTlvBatteryInfo`，只读展示 SOH、容量、电流、循环次数、版本等字段。
- 官方消息只读：接入系统消息、车辆消息、消息详情和消息设置的只读/低风险部分。
- 官方轨迹和围栏只读：先做列表、详情、地图展示，不做危险写入。
- 官方 NFC 列表只读：读取官方 `getNFCByQgj`，展示出厂钥匙、用户钥匙、槽位状态；继续禁用真实添加/删除。
- 官方车库/绑定信息只读：接入 `userCarList`、`getCarInfo` 等，补齐官方车辆详情、车架号、绑定状态、能力标识。

### 需要真机验证后再做

- 高级 QGJ 设置写入：自动锁车、HID、感应距离、电子龙头锁、边撑、坐垫、侧翻检测。
- NFC 真加钥匙、重命名、删除。
- BMS 校准、换电池、控制器激活限制相关操作。
- 多车系 BLE 写入，例如 BB、JLFBXW/C39。

### 暂不建议做

- 支付、充电桩真实交易、SIM 套餐购买、续费、保险、商城、金融。
- 真正 OTA。
- 官方推送生态的完整复刻。
- EasyConn 车机、导航投屏、离线地图、语音生态，除非后续明确把项目方向扩展到车机。

## 判断

当前复刻路线是正确的：先做核心控车和只读数据，再逐步补官方云端和低风险页面。真正危险的 ECU 写入、NFC 加钥匙、OTA 和支付交易暂时保持禁用，是当前阶段最稳妥的边界。

短期最有价值的补齐方向不是继续堆页面壳，而是把官方云端的只读数据接进来：BMS、电池、消息、轨迹、围栏、NFC 列表和官方车库详情。
